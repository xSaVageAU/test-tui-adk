Findings from reading `google.golang.org/adk/v2@v2.0.0` source directly (cached in the local Go module cache, `$GOMODCACHE/google.golang.org/adk/v2@v2.0.0`), gathered while building real multi-agent transfer into tui-testing's `internal/adk/client.go` — see [[project-direction-pivot]] for why this repo is now the real platform, not just a prototype.

**A `runner.Runner` has exactly one root agent, fixed for its lifetime.** `runner.Config.Agent agent.Agent` — doc comment literally says "Root agent which starts the execution." Plain `agent.Agent` interface, no special "root" type; root is purely positional (whatever you pass at `runner.New` time). Stored once (`rootAgent: cfg.Agent`) with no setter and no per-call override (`RunOption` only supports `WithStateDelta`). Confirmed in `runner/runner.go`.

**Four real ways multiple agents compose in ADK, each a different relationship:**

1. **Transfer / `SubAgents`** (`agent.Config.SubAgents`, `llmagent.Config.SubAgents`) — the model decides at runtime to hand off via an auto-injected `transfer_to_agent` tool. Control visibly moves: `session.Event.Author` changes to the new agent's name, and the runner's unexported `findAgentToRun` resumes at whoever last spoke across turns (walks session history backward for the last resolvable agent name, falls back to root). `DisallowTransferToParent`/`DisallowTransferToPeers` on `llmagent.Config` control direction. **No longer used in tui-testing** — this was the first working example, migrated to agent-as-tool on 2026-07-13 (see below).

2. **Agent-as-tool** (`tool/agenttool`, `agenttool.New(agent.Agent, *agenttool.Config) tool.Tool`) — wraps any agent as an ordinary callable tool for another agent. The caller stays in control, gets a result back, keeps its own voice — no visible transfer, no `Event.Author` change. `Config.SkipSummarization` controls whether the caller re-summarizes the wrapped agent's raw output before continuing. **This is what tui-testing uses today** — root `assistant` agent with `research`/`coder`/`planner` wrapped via `agenttool.New(agent, nil)` and added to its `Tools` list (see `internal/adk/client.go`'s `New`). `Event.Author` is now always the root's name, so the UI-side transfer-tracking (`ui.AgentSwitch`, `App.handleAgentSwitch`, the "→ transferred to X" transcript note) was removed as dead weight — one voice, always. Specialist tool results come back as `map[string]any{"result": "<full prose answer>"}` (confirmed in `agentTool.Run`'s final return), which needed two rendering changes in `internal/ui/chat.go`: `summarizeResult` now renders a single-string result bare (no "key=" prefix — it's prose, not data) instead of only handling list-shaped results, and `renderTool`'s status line now word-wraps with the gutter prefix repeated on every wrapped line (`renderToolStatusLine`) since a specialist's answer can run to a paragraph, not just a short list_files-style summary.

3. **Workflow agents** (`agent/workflowagents/{sequentialagent,parallelagent,loopagent}`) — fixed, programmer-defined topology (run children in order / concurrently / loop until `Escalate` or `MaxIterations`). No model routing judgment involved at all — for deterministic pipelines, not dynamic delegation.

4. **A2A** (`server/adka2a`) — cross-process/cross-codebase agent interop protocol (the Agent2Agent standard). Only relevant for agents outside your own tree/codebase; not a "multiple agents in my app" concern.

**Swapping the root agent itself has no live API**, but is achievable: build a new `*Runner` with a different root `agent.Agent`, reusing the *same* `session.Service` instance so conversation history survives — `findAgentToRun`'s author lookup just falls back to the new root for any historical speaker name it can't resolve in the new tree, which is the desired behavior (not an error case).

**Session/state is owned by `session.Service`, separate from the agent tree** — `session.InMemoryService()` today (wiped on process exit); `session/database` is a GORM-backed persistent alternative with the same interface. This is the thing that must be shared across a root-swap-by-reinit to preserve history.

**Decision (settled 2026-07-13):** agent-as-tool (#2 above) is the chosen delegation pattern going forward, not transfer (#1) — matches the user's old project's `api.md`, which described delegation as an entry in a tools list ("any other agent name for sub-agent delegation"), and matches the user's actual product intent: a single-agent application with one consistent voice, where "changing personality/capability" happens by swapping the *root* agent (see the root-swap-by-reinit technique above), not by mid-conversation transfer between personas. See [[agent-platform-vision]] for the fuller reasoning (one voice across every future surface — TUI/Discord/web — was the deciding factor). Transfer stays available as a mechanism if a future need for an actual multi-persona single conversation shows up, but isn't the default.

**Migration complete (2026-07-13).** Known, accepted costs from the switch, confirmed while researching `agentTool.Run` (`tool/agenttool/agent_tool.go`) and not yet addressed: each agent-as-tool call spins up a **fresh, disposable session** internally (no ongoing memory as that specialist across separate calls), and the wrapped agent's own token usage happens **inside** the opaque call — invisible to the parent's usage accounting (see the token-usage work in [[project-direction-pivot]]'s sibling history) unless explicit plumbing is added to surface it. Neither has caused a problem yet; revisit only if it does.

**Third cost, discovered as a real bug (2026-07-14): a sub-agent can never resolve a HITL confirmation, so any tool with `RequireConfirmation: true` is completely unusable by a sub-agent.** Traced through source, not just theorized: a confirmation-gated tool call inside a sub-agent's run produces a proper `toolconfirmation.FunctionCallName` event (`internal/llminternal/functions.go`'s `generateRequestConfirmationEvent`) — but that event can only ever be answered by the top-level app's own `Client.RespondToConfirmation`, which has no connection to a sub-agent's disposable internal runner (`agentTool.Run` creates its own throwaway `session.InMemoryService()` + runner per call, entirely opaque to the caller). `agentTool.Run`'s event loop only extracts `Text` parts from the final event; a stranded confirmation-request event has none, so it returns `map[string]any{}` — which the root then reports to the user as "the agent returned an empty response." Concretely hit when the user's newly-created `research` sub-agent (given `list_files` + `read_file`) called `list_files`, which had `RequireConfirmation: true` at the time.

Fix applied: removed `RequireConfirmation` from `list_files` entirely (`internal/adk/tools.go`) — it's read-only, so there was nothing to actually approve; the flag looks like leftover scaffolding from early HITL UI testing, not a real safety need. **`write_file` still has, and should keep, `RequireConfirmation: true`** (a genuinely destructive op, correctly gated for root's own direct use) — which means **`write_file` is currently unusable by any sub-agent** (a "coder" specialist with write access, the read→research/write→coder split the user has referenced repeatedly this project, cannot work as-is). No fix has been built for this yet; the honest options if it comes up again: (a) accept the limitation and never give sub-agents write access, (b) build some kind of confirmation pass-through from a sub-agent's disposable run back up to the top-level UI (nontrivial — agentTool.Run's internals aren't customizable from outside ADK), or (c) give a sub-agent a *different*, unconfirmed write tool variant, accepting that a sub-agent's writes then happen with no human approval step at all. None of these has been decided; flag this trade-off explicitly if the user asks for a write-capable sub-agent again rather than silently picking one.

**Sub-agents are now config-discovered, not hardcoded (2026-07-13, same day, follow-up).** The three specialists (research/coder/planner) built directly in `agents.go` were explicitly removed per user request ("essentially remove the sub-agents from being hardcoded at all... purely discoverable"). Current shape:
- `internal/adk/agents.go` — at the time this was written, only the **root** agent (`AgentName = "assistant"`) was still defined in code, per an explicit scope boundary the user drew ("only the root agent will stay hardcoded for now"). **That boundary was later reversed (2026-07-14, commit `0de9de9`) — root is now config-driven too**, via `agent.json`/`instruction.md` directly in appdir's root; see [[config-discovery-pattern]] for the current, authoritative shape (`AgentName` the const no longer exists — root's name is loaded at runtime). `buildRootAgent` still calls `loadSubAgentConfigs()`, builds whatever's found via `buildSubAgents`, wraps each in `agenttool.New`, and appends a generated "Available specialists: name — description" list to the root's instruction (`rootInstructionFor`) — empty list means that section is omitted entirely, so the root still works standalone.
- `internal/adk/subagents.go` — discovery lives here. `appdir.Path("subagents")` is created (empty) on every startup via `subAgentsDir()`; **nothing seeds it** — a fresh install has zero specialists, full stop. Each specialist is its own subdirectory named for it, e.g. `~/.tui-testing/subagents/research/`, containing exactly two files: `agent.json` (`name`, `description`, `tools` — a list of tool names resolved against a registry the caller builds, currently just `{"list_files": listFilesTool}`) and `instruction.md` (the agent's full instruction as plain prose/markdown, kept in its own file specifically so it reads/diffs/edits like text, not an escaped JSON string). A subdirectory missing either file, with bad JSON, or an empty instruction is a **hard load error**, not a silent skip — a half-authored agent should be visibly broken, not quietly absent.
- `researchAgentName`/`coderAgentName`/`plannerAgentName` and their instruction consts, and `defaultSubAgentConfigs`/seeding — all fully removed, not just unused. There is currently no default specialist shipped with the app at all; the user would need to hand-author `agent.json`+`instruction.md` files (or a future onboarding/scaffold command) to get any.
- This was a two-step iteration in one sitting: a first pass tried a single `agents.json`-per-specialist file seeded with the old three defaults (kept for backward compat), which the user then corrected — they wanted zero seeding and instruction split into its own markdown file. Don't re-introduce default-seeding or JSON-embedded instructions without being asked again.
- Verified via temp tests (written, run, deleted per usual discipline) both iterations: config load/seed/override, and later fresh-empty-discovery + hand-authored pickup + broken-folder hard-error. Committed as `74bc190`.

**How to apply:** if asked for a scaffold/template command ("give me a starting subagent"), or to make the root itself config-driven too, or to add more resolvable tool names to the registry — these are natural, well-understood next steps, not new research. If asked why there's no default specialist out of the box, this was deliberate, not an oversight.

**Frozen, 2026-07-17 — do not expand sub-agent work without the user explicitly re-opening it.** Confirmed by re-reading `agenttool.agentTool.Run` directly (`tool/agenttool/agent_tool.go`) that the context-gap the user suspected is real and structural, not fixable with a small patch: it spins up a disposable `session.InMemoryService()` per call and only extracts the **final event's text** — every intermediate tool call/result the sub-agent made is discarded before reaching root; root only ever passes the sub-agent explicit call args plus a filtered copy of session **state**, never the conversation transcript. The one ADK mechanism that *would* fix this — real `SubAgents` + `transfer_to_agent` (option 1 in the list above, already tried and deliberately abandoned on 2026-07-13 in favor of agent-as-tool) — shares the whole event stream, which would also make sub-agent HITL "just work" for free, but is a materially bigger pivot (transcript needs to show which agent is currently active, badges/tool lists become dynamic, transfer is a one-way handoff not a call/return). User's explicit call: don't dump the existing agent-as-tool code (it's small, ~175 lines, and still useful for its narrow bounded-utility-call case) and don't expand it (no sub-agent HITL fix, no write-capable sub-agents, no transfer-mode) until there's been a dedicated brainstorm on what a better multi-agent system should look like — reasoning being the project overall still lacks memory, a skill system, and transcript replay, so investing further in a half-designed sub-agent system now risks it staying half-baked "for ages." See [[project_tool-execution-targets]] — that future feature and any sub-agent redesign likely want a shared design, since both are "work happening somewhere other than the root agent directly."
