Built via a formal Plan-mode session (2 Explore agents + 1 Plan agent + direct verification of the agents' most load-bearing claims) after the user asked how to support MCP tools. Commit `70f6dc2`.

**Framing decision, made explicit before any design work:** the user's actual ask conflated two different things — connecting to standard/published MCP servers, and letting the agent "write its own tools on the fly." The second isn't an MCP primitive at all; it's arbitrary code execution (the agent writing a script, making it an MCP server, registering it, triggering a reload) — which turns out to already be *possible* today regardless of MCP, since `write_file` has zero path restriction. So it needed no bespoke engineering, just: (a) MCP registration being file-based config, and (b) a reload mechanism. Only the first (connecting to real MCP servers) was actually built as a distinct feature.

**Root-only, deliberately.** `agentFileConfig.MCPServers []string` exists on the shared root/sub-agent struct, but only `buildRootAgent` reads it — `buildSubAgents` never does, even if a sub-agent's `agent.json` has one (silently inert). Reason: ADK's `tool.ConfirmationProvider` (`func(toolName string, toolInput any) bool`) has no agent-identity parameter, so it can't replicate `gate.go`'s existing "sub-agent calls never need confirmation" rule (see [[project_adk-multi-agent-composition]]) — giving a sub-agent an MCP toolset with confirmation required would hit the exact same known "sub-agent can never resolve HITL" bug already documented there for `write_file`. Root-only sidesteps it entirely rather than accepting a known-broken interaction.

**The one critical, easy-to-miss ADK gotcha, found by a Plan agent and independently verified by directly reading the vendored source (`internal/llminternal/tools_processor.go:40-45`):** `llmagent.Config.Toolsets []tool.Toolset` (a field that existed, unused, before this — separate from the static `Tools []tool.Tool`) gets resolved via `Toolset.Tools(ctx)` **fresh on every single turn**, not cached at agent-build time, and **any error it returns fails the whole turn**, not just that toolset. Without a wrapper, one unreachable or misconfigured MCP server would permanently break the root agent's ability to respond to *anything*, forever, until fixed and reloaded — not a graceful "that tool's unavailable." Fixed with `resilientToolset` (`internal/adk/mcpservers.go`) — converts an inner `Tools(ctx)` error into `(nil, nil)`. Covered by `TestResilientToolsetSwallowsError`, not just asserted — this is exactly the kind of "unconfirmed library behavior" this project's testing-scope conventions say to actually verify, not trust from reading alone.

**Config shape:** `~/.tui-testing/mcpservers/<name>.toml` — one flat file per server (not a directory like sub-agents; no second "instruction" file needed), filename minus extension is the identity. Fields: `command`, `args`, `env` (merged onto `os.Environ()`). TOML for the same reason `settings.toml` is TOML — meant to be hand-copied out of a server's own README, not app-written. Malformed file = hard error at load (same severity as a broken sub-agent). Stdio/`mcp.CommandTransport` only for v1 — remote/authenticated MCP servers explicitly deferred (real, separate problem: token storage, OAuth flows).

**HITL:** every MCP tool requires confirmation outside full-auto, uniformly — no per-tool "this one's safe" distinction the way our own `read_file`/`list_files` get, since there's no way to know ahead of time which of a third-party server's tools are reads vs. real writes. `RequireConfirmationProvider` reads `settings.Load().Agent.PermissionMode`, same source `gate.go` already reads.

**`reload_agents` tool + the tool→UI bridge, genuinely new plumbing:** tools had zero prior precedent for reaching back into `internal/ui.App` or the running `*adk.Client` — confirmed no such mechanism existed anywhere in the codebase before this. Built as: a trivial HITL-gated tool (like `write_file`) that returns a sentinel result → `eventstream.go`'s `runStream` recognizes `FunctionResponse.Name == tools.ReloadAgentsToolName` and sends a *second* chunk, `ui.StreamChunk{ReloadRequested: true}`, alongside the normal `ToolResult` chunk → `App` latches a `reloadRequested` bool on seeing that chunk, but does **not** call `reloadBackend()` right then (it's still mid-turn; `reloadBackend()`'s own `turnInProgress()` guard would silently no-op) → deferred to `App.concludeTurn()` (turn.go), the same "turn genuinely ended" moment `popQueuedMessage()` already fires at (all four sites in `Update()`, from the message-queueing feature built earlier the same session — see [[feedback_concurrency-turn-queueing]]). `concludeTurn()` now replaces the bare `popQueuedMessage()` call at all four sites, batching in `reloadBackend()`'s cmd when the flag's set.

**User-confirmed live (same day):** asked the agent to repeat its instructions (no codeword present), added a codeword to `instruction.md` by hand, told the agent to reload, then asked again — it correctly picked up the new instruction after reload, no restart. Confirms the reload bridge works end-to-end for a human-driven config edit at least; full MCP tool-calling + HITL round-trip + the resilient-wrapper-against-a-real-broken-server case were set up for the user to verify live too (not independently confirmed back to me in detail — treat as "reload path confirmed," MCP tool-calling itself as "set up correctly, not explicitly re-confirmed in the follow-up message" if it matters later).

**How to apply:** if asked to extend MCP support to sub-agents, remote/authenticated transports, or a `/mcp` TUI management page (parallel to `/agents`), these are natural next steps, not new research — the config-discovery/HITL/reload machinery here should extend rather than be redesigned. If asked why sub-agents can't use MCP tools, point at the `ConfirmationProvider` agent-identity gap, not "not implemented yet."
