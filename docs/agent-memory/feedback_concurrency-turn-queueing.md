**The bug, found while just explaining how ctrl+c stop works (not while hunting for one):** `App.turnInProgress()` checked `a.streamChan != nil`. Two real gaps, not one: streamChan is never set at all in non-streaming (`Send`) mode, so a non-streaming turn was invisible to `turnInProgress()` for its *entire* duration — `/new`/`/sessions`/`/agents` reload could all race it. And even in streaming mode, streamChan only becomes non-nil once `streamStartMsg` lands, a whole async round trip after `dispatchToBackend` actually starts — a real window where a second send raced the first, corrupting the shared `streamingMsgIndex`/`turnCancel`. Fixed by checking `a.turnCancel != nil` instead — set synchronously the instant a turn starts, in both modes. **Lesson: when asked to explain how existing logic works, actually trace it rather than summarizing from memory — this bug surfaced only because the explanation required checking exactly which paths set which fields, and when.**

**Feature built on top: sendMessage queues instead of racing when a turn's already running; `/interrupt [prompt]` stops the current turn (or denies a pending HITL confirmation — no live call to cancel there) and, if given a prompt, replaces the queue rather than appending to it.**

**Two corrections from the user during this same feature, both about the same UI mechanism, worth internalizing together:**

1. First pass: a queued message got a `Queued bool` field, rendered inline in the transcript with a distinct "you · queued" label, plus a "Queued — will send..." system note, *and* a pinned strip at the bottom. User's correction: "that was close but not exactly what i meant... i dont want any of that to happen with queued messages. when we queue a message, it should only be pinned to the bottom... when the agent returns its final response, we should then remove the pinned message and send it in the chat as if the user only just sent it." **The actual want was more radical than "add a visual cue" — a queued message shouldn't exist in the transcript at all until it's genuinely sent.** This meant deleting the `ChatMessage.Queued` field, its style, and the inline label entirely — not layering a fix on top of the first attempt. Ended up simpler than the first pass, not more complex: `sendMessage`'s queue branch never appends to `a.messages`, only pushes text onto `App.queuedMessages`; `popQueuedMessage` (called from every real turn-conclusion site — success, error, stop, but not a HITL pause) appends the message for the first time and dispatches it via a new shared `sendNow` helper, indistinguishable from a message that was just typed.
2. The pin-at-the-bottom mechanism itself (`App.queuedPromptOverlay`, composited at `y = viewport height - overlay height` — distinct from the existing `stickyPromptOverlay`, which pins at the *top*, `y=0`, and only when scrolled past) was confirmed working correctly on the first pass and kept unchanged across the correction — the user's second message was scoped narrowly to "the transcript shouldn't also show it," not "the pin is wrong too." **Don't over-correct past what was actually flagged.**

**How to apply:** for any future "pending/in-flight" UI indicator, ask early whether the pending item should exist in the main content area at all before it's resolved, or *only* as a distinct pending-state indicator (a pin, a badge, a separate panel) — this user's default preference, shown twice now (see also [[feedback_badge-shading-restraint]] for the inline-styling side of the same instinct), is a clean binary state ("not visible yet" → "fully present, as if normal") over a three-state visual gradient ("normal" → "pending-styled" → "normal"). If a bug is found incidentally while doing something else (explaining code, reviewing a diff), trace it for real rather than reporting from memory of how it's "supposed to" work.
