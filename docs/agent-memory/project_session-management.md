On 2026-07-13 `/new` (start a fresh session) and `/sessions` (popup listing past sessions, pick one to switch to) were built on top of the persistent-sessions work. Scope was deliberately kept small ‚Äî confirmed with the user via two explicit choices rather than assumed:

1. **Switching sessions does not replay the transcript.** Picking a past session in `/sessions` clears the visible chat and drops a system note ("switched to sess_xxxx ‚Äî new messages continue that conversation, though earlier messages aren't shown here"). The backend genuinely still has full context for that session (ADK resolves it by ID transparently on the next message), but the UI doesn't reconstruct old messages into the transcript. Full replay was explicitly deferred ‚Äî it would mean re-deriving the whole rich `ChatMessage` model (tool call statuses, usage, etc.) from raw stored `session.Event` history, which is a separate, meaningfully bigger piece of work, not a small add-on.
2. **No delete support yet.** `session.Service.Delete` already exists and would be a small addition, but was deliberately left out of v1 to ship the smaller, more clearly-scoped version first.

**Technical notes from building this:**
- `session.Service.List` is metadata-only ‚Äî confirmed by reading `session/database/service.go`'s `List`, which queries only the `sessions` table, no event preload. There's no cheap way to show a content preview ("what was this session about") in the picker; only ID and last-updated time are available without a separate per-session fetch. The picker currently shows a shortened ID (reusing the header's `shortSessionID` helper) plus a relative time label ("5 min ago", "2 days ago", falling back to an absolute date past a week).
- `/new` and `/sessions` both refuse while a turn is mid-flight (`App.turnInProgress`, true when `streamChan != nil` or a HITL confirmation is pending) rather than trying to cancel and switch ‚Äî cleanly canceling an in-flight turn is its own not-yet-built feature (the "interrupt-and-redirect" idea from the Hermes-parity survey, see [[agent-platform-vision]]). Switching sessions mid-stream without this guard would risk a stale `streamChunkMsg` or a `RespondToConfirmation` call landing on the wrong (or freshly-reset) session.

**How to apply:** `/new`/`/sessions` still refuse outright while a turn is in progress rather than cancel+switch ‚Äî that specific upgrade hasn't been built. See [[feedback_concurrency-turn-queueing]] for what *was* built instead (2026-07-18): message queueing and `/interrupt`, which fixed a real bug in `turnInProgress` itself along the way.

**Session deletion was built 2026-07-18** ‚Äî DEL deletes the /sessions list's highlighted session, ctrl+DEL deletes every session, both behind a confirm popup (`paletteConfirmDeleteSession`). `Backend.DeleteSession(ctx, sessionID) error` is the new interface method. Real bug caught by the user and fixed same day: the confirm popup's "Delete" outcome used to reopen `/sessions` optimistically via the same pushMenuBack mechanism every other nested step uses ‚Äî but that reopen fired (and re-fetched) before the delete itself had actually finished, so deleting the *last* session (or all of them) landed back on a list that still showed the just-deleted stale data. Fixed by not reopening on the delete path at all: `confirmDeleteSession` calls `closeMenu()` directly (discarding the pending reopen, landing on the plain chat view), and the async result (`sessionsDeletedMsg`, now carrying a `remaining int` fetched fresh *after* the delete completes) is what decides in `Update` whether to reopen `/sessions` (`remaining > 0`) or leave the chat view showing (`remaining == 0`). Cancel is unaffected ‚Äî it never deletes anything, so the original pushMenuBack-reopen is still correct and fast for that path. Lesson for any future "delete then refresh a list" flow in this app: don't reuse the generic optimistic-reopen nav pattern when the reopened view's correctness depends on an async operation that hasn't resolved yet.

**Transcript replay was built 2026-07-17** ‚Äî no longer a gap. `Backend.GetTranscript(ctx, sessionID) ([]TranscriptEntry, error)` (`internal/ui/backend.go`) is a new minimal entry type (UserText/Text/ToolCall/ToolResult only ‚Äî no Reasoning field, deliberately dropped for v1 simplicity since it doesn't map cleanly onto the live Duration/Active badge tracking; revisit only if asked). `adk.Client.GetTranscript` (`internal/adk/eventstream.go`) walks `session.Service.Get(...).Session.Events().All()` ‚Äî confirmed by reading ADK's `runner/run_node.go` that only non-partial (fully-aggregated) events ever get persisted via `AppendEvent`, so replay needs none of `runStream`'s partial-delta/dedup handling, and the user's own messages show up as ordinary `Author == "user"` events (one per Send/Stream call, via `appendMessageToSession`). `ui.App.switchSession` now returns a `tea.Cmd` (was synchronous) that fetches the transcript async, landing via `transcriptLoadedMsg` (guarded against a stale reply if the user switches sessions again before the first fetch returns) and replayed with `App.replayTranscript`, which reuses the exact same `upsertToolMessage`/`completeToolMessage`/`dropEmptyStreamingPlaceholder` bookkeeping live streaming already uses, just applied synchronously in one pass instead of arriving one chunk at a time.

Verified against the app's own real sqlite session store (105 real sessions, via a throwaway `cmd/verifytranscript` program built, run, and deleted per usual discipline ‚Äî not hypothetical) rather than trusting the ADK source read alone, per [[feedback_testing-scope]]'s carve-out for unconfirmed library behavior. That verification surfaced a real wrinkle worth remembering: **a confirmation-gated tool call persists two `ToolResult` events under the same call ID** ‚Äî an auto-generated "requires confirmation" error placeholder first, then the real result once approved ‚Äî both sharing one ID. `completeToolMessage`'s plain overwrite-by-ID handles this correctly by construction (last write wins), confirmed and pinned down with a permanent test (`internal/ui/sessions_test.go`'s `TestReplayTranscriptDoubleResultSameID`), not just asserted from reading the code.
