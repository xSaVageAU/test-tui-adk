**Rule:** any new tool handler in `internal/adk/tools.go` that reads or writes a shared, order-sensitive resource (the filesystem, in-process state, anything where "did the previous call's effect land yet" matters) must serialize itself against every other handler touching that *same* resource — e.g. a `sync.Mutex` acquired for the handler's duration. Don't rely on the model calling things in the right order, and don't try to fix this with prompt engineering alone.

**Why:** confirmed by reading ADK v2 Go source directly (`internal/llminternal/base_flow.go`, `handleFunctionCalls`, ~line 1041): when a model response contains more than one function call, ADK spawns one goroutine per call and joins them with a `sync.WaitGroup` — genuinely concurrent execution, no ordering guarantee between them, and the function's own doc comment carries a TODO acknowledging this ("check feasibility of running tool.Run concurrently") — it's a known, deliberate design choice in the framework, not an edge case. Also checked: Gemini's `genai.ToolConfig.FunctionCallingConfig` only has `AUTO`/`ANY`/`NONE`/`VALIDATED` modes — nothing limits it to one call per turn, so there is no API-level switch to disable parallel calling.

The user hit this concretely in their prior project: a turn requesting `list_files` → `write_file` → `list_files` again (the second list meant to verify the write) as one batched/parallel call would race — the second `list_files` could run before `write_file` finished, report the new file missing, and confuse the agent into thinking the write failed. They correctly identified prompt engineering as a bandaid, not a fix, since it can't change ADK's dispatch mechanics even if the model's *intent* was sequential.

**Implemented 2026-07-14** (`internal/adk/toolgate.go`, commit `0bf6f1b`), once `read_file`/`write_file` were reintroduced (they'd been built once before, then explicitly reverted specifically because of this problem — see the read/write revert entry in [[adk-multi-agent-composition]]'s history). Design, refined through discussion with the user rather than a plain global mutex:

- **Resource-scoped, not tool-scoped or session-wide.** Each tool call declares what it touches via a `resourceRef{Key, Write, Recursive}` (a normalized absolute path, whether it's a write, and whether it's a directory scope like `list_files`). Two refs conflict only if their keys overlap (exact match, or containment when one side is `Recursive`) **and** at least one side is a write — the user's own spec: five parallel `write_file` calls on five different files never block each other; `write_file(X)` + `read_file(X)` or `write_file(X)` + `list_files(dir containing X)` do.
- **A pending HITL confirmation keeps holding its resource lock**, not just mutual exclusion during actual execution — this is the specific fix for the bug the user hit in their old project (HITL calls appeared to "execute in order" while a non-HITL sibling raced ahead and completed first): `gatedTool.Run` acquires the gate, and if the wrapped tool returns `tool.ErrConfirmationRequired`, it parks the release func (keyed by `agent.Context.FunctionCallID()`, which stays stable across the pause/resume boundary — confirmed by reading `handleFunctionCalls`' `toolConfirmations[fnCall.ID]` lookup) instead of releasing, so a conflicting sibling call blocks until the human actually answers.
- **Per-ADK-session**, keyed by `agent.Context.SessionID()`, via a package-level `map[string]*sessionGate` (never evicted — cheap enough per-session that eviction wasn't worth building for a single-user app).
- **Known, accepted scope boundary**: a specialist consulted via agent-as-tool runs its own tool calls in a *fresh, disposable* session (`agenttool.Run` creates one per call), so the gate doesn't catch root-vs-specialist or specialist-vs-specialist races on the same file — only races among calls the same session directly makes. Documented in `toolgate.go`'s own doc comment; revisit only if this combination actually causes a problem.
- Wrapping mechanism: `gated(t tool.Tool, resources func(map[string]any) []resourceRef) tool.Tool` — since `tool.Tool` embedding only promotes the narrow 3-method interface (not `Run`/`Declaration`/`ProcessRequest`, which live on ADK's *internal*, unimportable `toolinternal.FunctionTool`), `gatedTool` restates that shape locally as `funcTool` and type-asserts the wrapped tool against it at construction (panics if it doesn't hold — a programmer error, not a runtime condition). Verified this assertion actually succeeds at runtime (not just compiles) via a temp test that constructed all three tools for real.
- A tool whose `resources` func returns nil/empty (no path-shaped args) runs completely unguarded — opt-in, not opt-out, by design.

**How to apply going forward:** any new filesystem- or shared-state-touching tool must supply a `resources` func and be wrapped with `gated()` in `tools.go`, same shape as `listFilesResources`/`readFileResources`/`writeFileResources`. Don't reach for a plain global mutex — the resource-scoped design is deliberately more permissive (only conflicting calls serialize) and this is the second time in this project a naive full-serialization approach was considered and rejected in favor of it.

**Critical follow-up bug, found and fixed 2026-07-14 (same day, later): the gate was silently inert in the real running app the entire time it existed, and only ever exercised by isolated unit tests.** Root cause: `gatedTool` (and later `confirmGatedTool` — see below) embed the wrapped tool's `funcTool` interface and only override `Run`, leaving `ProcessRequest` to promote through untouched. But `functiontool.ProcessRequest` internally calls ADK's unexported `toolutils.PackTool(req, f)`, which does `req.Tools[f.Name()] = f` — **`f` there is the method's *receiver*, i.e. whatever concrete value `ProcessRequest` was actually called on**. Since promotion means `ProcessRequest` runs with the *innermost raw tool* as its receiver (not the wrapper), `req.Tools` — the exact map ADK's `handleFunctionCalls` later turns into its execution dict — ends up holding the unwrapped tool. `Name()`/`Description()`/`Declaration()` all still promoted and worked fine (which is exactly why this was invisible in every test and in normal use), but `Run()` was **never actually called on any wrapper** — only on the raw `functiontool` underneath, silently skipping both resource-conflict serialization and (once built) permission-mode confirmation gating.

Caught only because the user reported `write_file` never asking for confirmation even in normal mode after a fresh restart — investigated with a real live end-to-end test (`client.Stream` against the actual saved API key, watching for a `Confirmation` chunk) rather than more source-reading, since source-reading alone had already produced a plausible-but-wrong theory (a `ctx.Agent().Name()` mismatch — see below) that didn't hold up.

**Fix:** both `gatedTool` and `confirmGatedTool` now explicitly override `ProcessRequest` too — call the wrapped tool's `ProcessRequest` first (still does the real work of populating `req.Config`/schema), then overwrite `req.Tools[name]` to point at *itself* instead. Verified live, twice: normal mode now genuinely produces a `Confirmation` chunk, full-auto mode genuinely skips it and the tool still runs.

**Second bug found in the same investigation:** the sub-agent-detection mechanism (meant to auto-accept confirmations for a specialist, which can never resolve one) originally used `ctx.Agent().Name()`. This panics inside any tool's `Run` — confirmed via a live crash, not just source-reading — because `agent.NewToolContext` wraps everything in an internal `toolContextWrapper` whose `Agent()` is deliberately stubbed to log "Agent() is not supported for tool context" and return `nil`. The correct, actually-supported call is **`ctx.AgentName() string`** (forwarded properly by the same wrapper) — use that, never `ctx.Agent()`, from inside a tool's `Run`.

**How to apply:** any future `tool.Tool` wrapper in this codebase (or anything similar) MUST override `ProcessRequest` and re-register itself in `req.Tools`, not just `Run` — promoting it through an embedded interface is not enough, and the failure mode (tool still appears to work, schema/name still correct, only real dispatch is silently bypassed) is exactly the kind of thing that survives casual testing. When a live/reported behavior contradicts a source-grounded theory, trust the live result and re-derive — don't assume the source-reading was right just because it was careful; test the *specific* thing that's contradicted, not everything.
