On 2026-07-13, in the same sitting, two unrelated features were converted from Go-literal hardcoding to the same config-discovery shape, at the user's explicit request each time — this is now the app's standard way to make something user-extensible, not a one-off:

1. **Sub-agents** (see [[adk-multi-agent-composition]]) — root agent stays hardcoded in `internal/adk/agents.go`; every specialist is discovered from `~/.tui-testing/subagents/<name>/{agent.json, instruction.md}`. Nothing is seeded — a fresh install has zero specialists.
2. **Themes** — built-ins (`Mono`, `Vibrant`, `Nord`, `Dracula`, `Gruvbox`, `Tokyo Night`, `Catppuccin Mocha`) moved from `internal/theme/presets.go` (deleted) to JSON files under `internal/theme/defaults/`, embedded into the binary via `go:embed` (`internal/theme/load.go`). `theme.Load()` merges those with anything in `~/.tui-testing/themes/*.json`; a custom theme sharing a built-in's name replaces it in place (same cycle position) rather than duplicating.

**The shape, generalized, for the next time this comes up:**
- A struct that's pure data (already true of both `subAgentConfig`/`Theme` before this — the win here is *loading* it from files, not a data-model change).
- Codebase-shipped defaults live as literal config files in the repo, embedded via `go:embed` so the binary stays single-file/portable. Format is JSON for anything machine-written or edited through an interface (agent.json, theme files, sub-agent configs) — `settings.json` was the one exception, switched to TOML (`settings.toml`) on 2026-07-18 specifically because it's the one file meant to be comfortable to hand-edit; see the dated entry below before assuming "add a YAML/TOML dependency" is off the table generally — it's scoped to config surfaces meant for hand-editing, not a blanket preference.
- A matching directory under `appdir.Path(...)` (a new named subdirectory per feature — `"subagents"`, `"themes"`) is where a user's own additions/overrides go, created empty on first access if missing.
- **Error-handling severity differs by stakes, decided deliberately each time, not by rote consistency:** sub-agent configs fail *loudly* (a whole broken subdirectory blocks that agent from existing, with a clear error) because a half-built specialist silently vanishing would confuse someone actively testing it. Theme configs fail *quietly* (a broken custom theme file is skipped, the rest of the app still starts) because theming is cosmetic and startup must never be blocked by a color typo. When adding the next config-discovered feature, make this call explicitly rather than copying one or the other by default — ask "should this be invisible-if-broken or loud-if-broken" on its own merits.
- A broken **embedded default** (not a user file) is treated as a build-time bug, not a runtime condition — themes uses `panic` for that case specifically (`mustLoadDefault`), since a shipped-broken default would fail identically on every single run and should be caught in development, not degraded around.

**Third instance, a variant (2026-07-14): `settings.json`** (`internal/settings`, commit `29cc6c6`) — not pure discovery like the two above, but a deliberate *mixed-ownership* single file: a `ui` section (`highlightUser`/`streamReplies`/`hitlMode`) the app itself writes automatically whenever `/settings` changes one, versus an `agent` section (root's `provider`/`model`) that's hand-edited by the user exactly like a sub-agent's `agent.json` and never written by the app. The two sections coexist in one file without stepping on each other because every write is load-modify-save on the whole `Settings` struct (`App.persistSettings`) — never a partial/UI-only struct that would zero out the hand-edited section. Same file also introduced per-agent model selection: `internal/adk/models.go`'s `buildModel(ctx, provider, model, apiKey)` resolves an agent's model, defaulting empty fields to `gemini`/`gemini-3.1-flash-lite` and erroring immediately (not silently falling back) on any other provider name, since only Gemini is actually implemented. A sub-agent's own `agent.json` can now set `provider`/`model` too, overriding root's; omitted inherits root's already-built model object rather than rebuilding an identical one.

**Root agent became config-driven too (2026-07-14, commit `0de9de9`) — reverses an earlier explicit decision.** Originally the user said "only the root agent will stay hardcoded for now" (see [[adk-multi-agent-composition]]); after hitting confusion where the agent only knew about `list_files` (zero sub-agents were ever configured, and root itself was hardcoded to one tool), the user asked for root to get all three tools *and* become config-file based like sub-agents. Now: a single `agent.json`/`instruction.md` pair lives directly in appdir's root (not a subdirectory — there's only one root). Key divergence from sub-agents' policy: since root is mandatory (the app can't run without one), a **missing** file self-heals by seeding a working default (`internal/adk/rootagent.go`'s `loadRootAgentConfig`/`seedIfMissing`) rather than being absent-by-design; **malformed** content still fails loudly, same as sub-agents. `settings.json`'s `agent` section (added earlier the same day) was removed as part of this — it became redundant with root's own `agent.json`, and rather than keep two places that could each claim to set the root's provider/model, the newer, more consistent design won. `settings.json` is UI-toggle-only again. `RootAgentName()` reads just the name without needing an API key, so the header shows the right value even pre-connection.

**appdir layout, as of `0de9de9` (2026-07-14):** `~/.tui-testing/` root holds only plainly human-editable files/dirs — `agent.json`, `instruction.md`, `settings.json`, `subagents/`, `themes/`. Generated or secret data (`data.db`, `credentials.json`) moved into a `data/` subdirectory, via a shared `dataPath()` helper in `internal/adk` (wraps `appdir.Path("data", ...)` + `os.MkdirAll`). If a future feature needs to persist something, ask which bucket it belongs in before picking a location.

**Fourth instance (2026-07-18): MCP servers.** `~/.tui-testing/mcpservers/<name>.toml`, one flat file per server (no directory — unlike sub-agents, no second "instruction" file needed), filename-is-identity same as sub-agent directories. Malformed = hard error, same severity choice as sub-agents. See [[project_mcp-client-support]] for the full design (root-only, HITL, the reload mechanism, and a critical ADK per-turn-toolset-resolution gotcha found and fixed along the way).

**How to apply:** if the user proposes making another hardcoded thing (tools? prompts? key bindings?) "config-based" or "user-extensible," this is the reference shape to reuse — don't re-derive the design from scratch, but do re-derive the error-severity call for that specific feature, and consider explicitly whether it's mandatory-singleton (self-heal missing files, like root) or optional-plural (silently absent is fine, like sub-agents/themes). If a future feature needs both app-managed and hand-edited data in the same conceptual file, the settings.json load-modify-save pattern is still the reference for that, not a reason to split into two files — just don't let it grow a section that duplicates something an agent.json now owns.

**Credentials generalized to multi-provider, OpenRouter prep started (2026-07-14, later the same day).** Per-agent `provider`/`model` in agent.json (described above) was already fully wired by the time the user asked "can we make it so the model/provider is specified per-agent" — turned out to already be done from the settings.json/root-config-driven work earlier that day; nothing new needed there. What genuinely was missing: `credentials.go`'s `SaveAPIKey`/`LoadAPIKey` were hardcoded to the Gemini entry regardless of what provider a key was actually for — the on-disk format (`Providers map[string]map[string]string`) already supported multiple providers, the Go functions just didn't expose that. Generalized both to take a `provider` param, exported `ProviderGemini`/`ProviderOpenRouter` (was unexported `providerGemini`) so `main.go`'s `/key` flow can name which provider it's saving for. `buildModel` (`internal/adk/models.go`) now resolves its own API key per-provider via `LoadAPIKey(provider)` when no override is given, instead of requiring a single key threaded in from `main.go`/`New` — `geminiOverride(provider, apiKey)` is the seam: main.go's key is only ever a Gemini key, so it's only used as an override when the agent being built is actually configured for Gemini; anything else (e.g. a sub-agent set to `"provider": "openrouter"`) now correctly resolves its own stored key instead of silently reusing Gemini's. `adk.New`'s old up-front "no API key given" guard was removed — that check now happens inside `buildModel`, scoped to whichever provider is actually needed, so a root/sub-agent not configured for Gemini no longer requires a Gemini key to exist at all.

**Caught immediately by the user, same conversation:** I'd claimed per-agent `provider`/`model` was "already fully wired" based on reading `buildModel`'s resolution logic — but never actually checked what a freshly seeded `agent.json` contains on disk. `agentFileConfig.Provider`/`Model` are `json:",omitempty"`, and `rootagent.go`'s `seedIfMissing` built the seed literal without setting them, so they silently vanished from the marshaled file — a fresh install's `agent.json` never showed the fields at all, even though typing them in by hand would've worked. Fixed by seeding `Provider: ProviderGemini, Model: DefaultModelName` explicitly so the fields are visible and editable from the very first run, matching the file's whole purpose. **Lesson:** when a memory or prior code says a config field is "wired," verify what the seed/self-heal path actually emits to disk, not just that the resolution/read path handles it correctly — `omitempty` (or any zero-value-drops-silently serialization) makes "the Go code supports it" and "the user can discover it" two different, separately-checkable claims.

**OpenRouter provider actually implemented (2026-07-14, immediately after the prep above).** `internal/adk/openrouter/` — a from-scratch `model.LLM` (`openrouter.go` translation logic, `wire.go` plain OpenAI-compatible JSON types, `aggregator.go` streaming-delta accumulator), wired into `buildModel`'s `ProviderOpenRouter` case (`openrouter.NewModel(modelName, apiKey)`, no error return — nothing to fail at construction, just an HTTP client). Key design points, in case this needs revisiting:
- **Message routing is part-type-driven, not `Content.Role`-driven**: a `genai.Part.FunctionCall` always becomes part of an assistant `tool_calls` array, a `FunctionResponse` always becomes its own `role:"tool"` message — regardless of what role genai's own bookkeeping tagged the surrounding `Content` with. This sidesteps ever needing to know/trust what role ADK put on a tool-result Content.
- **Streaming aggregation deliberately mirrors ADK's own vendored `internal/llminternal/stream_aggregator.go`** (read directly, not guessed): emit one `Partial:true` `model.LLMResponse` per text delta immediately, tool-call argument fragments accumulate silently (per-index, keyed by the streaming delta's own `index` field — required since OpenAI-style parallel tool calls interleave fragments by index) until the stream ends, then exactly one final response with `Partial` left `false` (the zero value — confirmed upstream's own `Close()` never sets `TurnComplete` either, so this doesn't set it, matching rather than guessing at a field the reference implementation itself leaves unset).
- **`DefaultModelName` (`gemini-3.1-flash-lite`) only applies to the Gemini case now** — an OpenRouter config with no `"model"` set is a clear config error ("requires an explicit model, e.g. openai/gpt-4o-mini"), not a silent attempt to run a Gemini-shaped model name against OpenRouter's API.
- Schema translation (`schemaToJSONSchema`) only covers what this app's own tools actually emit (object/string/array, properties/items/required/description, uppercase→lowercase `type`) — not genai.Schema's full surface (anyOf, numeric bounds, patterns), since nothing in this codebase produces those.
- **Tested, but not against the real OpenRouter API** — no OpenRouter key was available/offered, and spending the user's credits without asking felt presumptuous. Instead: `internal/adk/openrouter/openrouter_test.go` uses `httptest.Server` (via a package-level `chatCompletionsURL` var, swapped for the test's duration) to verify non-streaming text, non-streaming tool calls, request-side tool-call/tool-response round-tripping (the two messages must share a `tool_call_id`, which is the actual correctness risk in `contentsToMessages`), and streaming text+tool-call aggregation (interleaved argument fragments, partial-vs-final response shape). This was a deliberate escalation past build+vet — see [[feedback-testing-scope]] — because this is genuinely new, hand-rolled protocol-translation code with no framework backing it, not routine wiring. Still open: an actual live smoke test once the user has an OpenRouter key and wants to point an agent at it.

**`/key` made provider-aware, `/agents` brought back (2026-07-14, commit `4eac735`).** `/key` now opens a provider picker (Gemini/OpenRouter) before the masked key field, saving under the right `credentials.json` slot; still validates before persisting. `/agents` lists root + every discovered sub-agent, and lets you change either's Provider (a picker) or Model (a prefilled text field) — saves straight to that agent's `agent.json` via new `internal/adk/agentconfig.go` (`ListAgentConfigs`/`SetAgentProvider`/`SetAgentModel`, id `""` for root or a sub-agent's directory name), then immediately reloads the running backend by calling the same `newBackend` factory `/key` uses with `("", "")` — no override key, so every agent resolves its own saved credentials fresh. `adk.New`/`buildRootAgent`/`buildSubAgents` were generalized from a bare Gemini-only `apiKey` param to `(provider, apiKey)`, and `geminiOverride` became `keyOverride(configProvider, callerProvider, apiKey)` — only used as an override when the two providers actually match. `internal/ui` still never imports `internal/adk` directly: `AppConfig` gained `ListAgents`/`SetAgentProvider`/`SetAgentModel` closures, same bridging pattern as `NewBackend`, wired from `main.go`.

**`settings.json` → `settings.toml`, `/settings` split into TUI/Agent categories (2026-07-17/18, commits `16c13e6`/`5d7e67b`).** Two related but separable changes, both prompted by the user wanting to prep for a larger config surface:
1. **`/settings` is now a two-level menu** — a category picker (paletteSettings) opening either "TUI Settings" (paletteSettingsTUI: highlight, stream, hitl style, verbose, reasoning, popup size, tool preview lines) or "Agent Settings" (paletteSettingsAgent: currently just permission mode). `settings.Settings` gained a top-level `Agent AgentSettings` section alongside `UI UISettings`, with `PermissionMode` moved out of `UISettings` into it. Incidental UX improvement from the nesting: toggling a setting now returns to that submenu (via the same pushMenuBack mechanism `/agents`' nested steps use) instead of closing the whole popup like the old flat menu did — costs losing cursor position (list reopens at the top each time), not fixed since it'd need the same in-place-update mechanism `/agents`' Tools checklist has, judged not worth it yet.
2. **JSON → TOML, `github.com/pelletier/go-toml/v2`.** User's reasoning, worth remembering for any *other* config file that comes up: agent.json/theme files are fine as JSON because they're written by the root agent or edited through an interface, never really hand-edited — `settings.json` was different, explicitly meant to be comfortable to edit by hand, and "editing json manually can be daunting." Chose TOML over YAML on my recommendation (purpose-built for config, avoids YAML's Norway-problem/indentation footguns) — user had no format preference of their own, deferred to that reasoning. Key names kept identical (`highlightUser`, `permissionMode`, etc.) rather than converted to snake_case, purely so a one-time migration could reuse the exact same `Settings`/`UISettings`/`AgentSettings` struct for both `json.Unmarshal` (encoding/json case-insensitively falls back to matching by field name when there's no `json` tag) and the new `toml.Unmarshal` — avoided writing a second, parallel struct just for the migration.
3. **One-time migration, then deleted.** `loadLegacyJSON()` in `Load()` recovered a pre-TOML `settings.json` (both the very old shape with `permissionMode` nested under `ui`, and the brief split-but-still-JSON intermediate shape) into `settings.toml` on first launch after the update. Verified end-to-end against a *copy* of the user's real on-disk `settings.json` (not just synthetic test data) before considering it done. Once the user confirmed their real settings migrated correctly (`~/.tui-testing/settings.toml` now holds their real `verboseTools=true, workingAnim=Equalizer, permissionMode=full-auto`, etc.), they explicitly asked to delete the migration code rather than keep it as permanent dead weight — `loadLegacyJSON`, `legacySettingsFileName`, and its three dedicated tests are gone; `Load()` is back to a plain single-format read. **If this pattern needs to repeat for some other file's format/shape migration in the future: write the migration, verify it against real data, ship it, then remove it once the one person who needed it has actually run it once** — don't leave migration shims sitting around indefinitely "just in case" on what is currently a single-user app.

**User-confirmed live (same day, right after this shipped): OpenRouter resolves bare model names leniently — no `vendor/model` prefix required.** User set the root agent's model to plain `gemini-3.1-flash-lite` via the new `/agents` UI (with `provider` set to `openrouter`) and it genuinely worked — confirmed via OpenRouter's own activity-log dashboard showing real usage against the intended model, not a silently-still-Gemini fallback (a real alternative explanation that was checked and ruled out: `reloadBackend`'s `keySetMsg` handler only swaps `a.backend` on success, so a failed reload would have silently kept the old Gemini connection alive — that's *not* what happened here). Also tried `gpt-5.6-luna`/`claude` models both with and without an `openai/`-style prefix — identical behavior either way. This package's request translation (`internal/adk/openrouter`) never validates or normalizes the model string — `chatRequest.Model` is whatever `agent.json` says, passed straight through — so this is purely OpenRouter's own server-side matching being more permissive than the `vendor/model` convention implies. The `/agents` model-field placeholder ("e.g. openai/gpt-4o-mini") is worth keeping as a disambiguation convention, but isn't a hard requirement — don't tell a future user a prefix is mandatory.
