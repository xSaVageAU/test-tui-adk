**STATUS UPDATE 2026-07-18: the SSH target is built and verified.** The abstraction this memory anticipated now exists — a `Target` interface in `internal/adk/tools/target.go` (Run/StartBackground/ReadFile/WriteFile/Stat/ReadDir/Open/Walk/Close), with `localTarget` (target_local.go, the old os/exec behavior verbatim) and `sshTarget` (target_ssh.go, persistent ssh.Client + sftp.Client). Every tool goes through `target()` instead of os/exec directly. Config is `settings.AgentSettings.Target` (type "host"/"ssh" + SSH block: host/port/user/keyPath/knownHosts/insecureSkipHostKey). Wired via `tools.ConfigureTarget()`/`CloseTarget()` re-exported through adk, called from main.go at startup/exit. Decisions made: persistent connection + per-command exec sessions (no stateful cwd); SFTP walk for grep/glob (correctness over speed); SSH background procs deferred (StartBackground errors on ssh). Verified in-process against a loopback ssh+sftp server (auth, exec+exit-code, all file ops). A /settings → Agent Settings → "Tool execution target" row cycles host↔ssh at runtime (reconnects, reverts to host on SSH failure); SSH host/user/key are still hand-edited in settings.toml. Fixed a latent bug: settings.Load() used to reset the whole Agent section to defaults when permissionMode was empty, which would wipe a hand-added [agent.target] — now field-level. NOT done: passphrase keys / ssh-agent auth; remote background lifecycle; editing SSH connection params in-UI. See [[project_coding-toolset]].

**Real-world stress test 2026-07-18** (agent built a 14-file Go webserver on the LXC target, wrote /root/website/report.md): fileops over SFTP (write/read/edit/grep/glob/list) were FLAWLESS on real remote Debian — strong validation of the Target refactor. run_shell over SSH is the weak link, and it's all the deferred SSH-background gap: no background:true → forces brittle `nohup &` hacks that hang because the ssh exec channel stays open on inherited stdin (session.Wait never returns → our timeout SIGKILLs the server). The reported "kill+restart returns 143" was NOT our bug (agent's `pgrep -f server` matched its own shell). Takeaway: **SSH background lifecycle (StartBackground/shell_output/stop_shell over ssh: setsid + full fd redirect, track remote PID, tail log via sftp, kill by PID) is the priority follow-up** — "run a server and check on it" is a core workflow and currently painful over SSH.

**SSH background lifecycle BUILT + verified against the LXC 2026-07-18.** Local/remote bg procs now share a `bgHandle` interface (bgproc.go); sshBgProc (bgproc_ssh.go) launches via a runner script written over SFTP (no shell-quoting hazard) + `setsid sh script </dev/null >log 2>&1 & echo $!` (detaches, returns PID immediately — fixes the report's hang), exit code captured via an `EXIT` trap (survives explicit `exit N`), shell_output tails the log over SFTP from a byte cursor, stop_shell kills the group (`kill -TERM/-KILL -PID`). Verified live: incremental output, running status, group-kill leaves NO orphans (sleep children died too), exit-code=7 captured. Remote bg procs survive TUI exit (stopOnAppExit=false — independent like nohup; only local ones are killed on shutdown). Known minor gaps: /tmp/tui-agent-bg/*.{sh,log,exit} accumulate on remote (in /tmp, cleared on reboot); a remote proc from a previous run isn't recoverable after restart (id only in in-memory registry); assumes `setsid` present (fine on Debian).

Below is the ORIGINAL brainstorm (2026-07-17), kept for the still-unbuilt target kinds (MCP / remote-agent / container):

Raised 2026-07-17, explicitly flagged for the future — not started, no design committed, do not begin implementing without the user raising it again.

**The idea:** give tools a pluggable "execution target" instead of always running against the local machine the TUI process itself is on — effectively sandboxing/relocating tool execution. Motivating example: an SSH target can't have the agent binary running on it, so file-editing tools would need an SSH-equivalent implementation (e.g. SFTP: pull the file into memory, edit it there, push it back) rather than local `os` calls.

**Candidate mechanisms the user named (none chosen yet):**
- Per-tool remote-equivalent implementations (e.g. SFTP-backed read/write for an SSH target).
- Tools exposed as an MCP server, reachable from wherever the agent runs.
- A "remote agent" — a separate agent process actually running on the target host, talked to rather than executing directly.
- Container sandboxing — Docker/runc, or gVisor/runsc for stronger isolation — running the existing tool implementations inside an isolated environment.

**Why this matters now even though it's not being built now:** it implies tools need a real abstraction layer (same logical tool, e.g. `write_file`, backed by different implementations chosen per target) rather than each tool hard-assuming local execution. This directly intersects [[project_adk-multi-agent-composition]]'s standing "don't bloat the tool list / system-prompt context" constraint — deciding which tools to add and how they're described should keep this future abstraction in mind so early tools aren't built in a way that has to be unwound later, even though the target-selection machinery itself is deferred.
